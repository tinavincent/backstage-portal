# Core Principles

These principles apply to every project created from this template.

They should change rarely.

---

## User Value First

Build things because they help a user do something meaningful.

Do not build features only because they are technically interesting.

---

## Small Iterations

Prefer small, visible progress over large plans.

Each iteration should be easy to understand, implement and validate.

---

## Discovery Before Implementation

Challenge assumptions before building.

Use the smallest experiment that can reduce uncertainty.

---

## Product Before Technology

Start with the problem, user and value.

Choose technology after the product need is clear.

---

## Simplicity Over Cleverness

Prefer understandable solutions over clever solutions.

Avoid abstractions, frameworks and architecture that the product does not need yet.

---

## Workflow Before Dashboard

Most useful apps help a user move through a task.

Prefer clear flows, next actions and contextual information over dashboards full of metrics.

---

## Visible Outcomes

A change only creates value when the user can see, understand or verify the result.

Automation should always make its outcome visible.

---

## Reuse Before Building

Before adding something new, check whether an existing pattern, component, data structure or tool can solve the problem.

---

## Scope Discipline

Work on one feature, problem or decision at a time.

Split work when it becomes hard to explain or validate.

---

## Documentation Serves Work

Documentation should help the project move forward.

Avoid adding process or documents that do not improve decisions, implementation or maintainability.
